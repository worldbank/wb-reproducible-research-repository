# Guidelines for Reviewing the Data Availability Statement (DAS)

These guidelines are for reproducibility reviewers assessing whether the Data Availability Statement (DAS) in a README is complete. The DAS can appear as a standalone section within the README or as a separate file; either is acceptable. The goal is to ensure that a third party could locate, access, and understand the data used in the paper without any additional information from the authors.

---

## Step 1: Check that a DAS exists

The package must include a DAS. If there is no DAS and no data documentation of any kind, return the package to the authors before proceeding further.


---

## Step 2: Check the opening statement and Statement of Rights

### Opening statement [Recommended]

The DAS should begin with an overall categorization of the data used. If it is present, check that it correctly reflects the data in the package, i.e., that the accessibility level claimed (e.g., "all data are publicly available") actually matches the datasets used. The statement should indicate one of the following:

- All data are publicly available
- Some data cannot be made publicly available
- No data can be made publicly available
### Statement about Rights [Required]

A [Statement about Rights](https://github.com/worldbank/wb-reproducible-research-repository/blob/update_readme/resources/README_Template.md#statement-about-rights) is required in every DAS. It should confirm that the authors had legitimate access to the data and address redistribution rights, i.e., whether they have permission to share the data, or, if not, why not and under what terms others can access it.

If this statement is missing, flag it as a required addition. 

---

## Step 3: Check that every input dataset is documented

### 3.a Identify which datasets need to be documented

Every dataset that is an **input** to the code, meaning it is not created by the code itself, must be documented in the DAS. This includes datasets in intermediate data folders if they are inputs rather than outputs.

If it is not obvious from the folder structure which files are inputs versus outputs, use the [Data Hash Tool](https://github.com/worldbank/wb-reproducible-research-repository/tree/main/resources/generate_hash) after running the code: any file that was **not modified** after the code ran is an input and needs to be documented. 
>Note that some files may show timestamps from the authors' system that appear to be in the future. Exercise caution when using timestamps to distinguish inputs from outputs.

### 3.b Check the information provided for each dataset
Once you have identified all input datasets, check whether each one is documented in the DAS with the required fields.

#### General requirements (all datasets)

 
| Field | Required | Notes |
|---|---|---|
| Filename(s) | Required | Exact file name(s) as they appear in the package |
| Source / data provider | Required | Name of the institution or organization |
| URL | Required if available | Direct download link where the data can be accessed |
| Access date | Required | Month and year is sufficient |
| Full data citation | Recommended | Not strictly required, but strongly recommended. If missing, it can be generated using the [README Generator](https://dime-worldbank.github.io/wb-reproducible-research-repository-automation/) or following the [Social Science Data Editors' guidance on citing data](https://social-science-data-editors.github.io/guidance/Guidance/addtl-data-citation-guidance.html). A URL alone is not a citation. |
| License / redistribution terms | Required if restricted; recommended otherwise | It is your responsibility as the reviewer to check the terms of use for each dataset and confirm that each data is tagged to the right terms. The terms of use are usually found on the data provider's website - look for a "License," "Terms of Use," or "Data Access" page. If you are unsure how to interpret the terms, check with the coordinator. |
| Access instructions | Required if not directly downloadable | Required if data requires a request, subscription, or point-and-click selection; specify time period, variables, or subpopulation filters applied. |

#### Additional requirements for World Bank-owned data
 
If any data was generated by a World Bank team (including the authors themselves), it must be deposited in one of the Bank's official repositories:
 
- [Microdata Library (MDL)](https://microdatalib.worldbank.org/index.php/data-deposit) for survey and microdata
- [Development Data Hub (DDH)](https://datacatalog.worldbank.org/int/data/add) for geospatial, administrative, time series, and other data

If the data has not yet been deposited, inform the authors that this is required. They have the option to deposit as internal/restricted, publication does not mean the data must be made public. 

If the data is forthcoming or pending deposit, you can proceed with verification but must return to update the metadata entry once the data is published.
 
**Exception:** Data deposit is not required if the dataset was created by the authors specifically for this study using public sources, and has no value or use outside of the study itself.
 
#### Additional requirements for third-party unpublished or restricted data

If any dataset is restricted or confidential, the DAS must also include enough information for a reader to understand how they could request the same data along with a citation.

## Step 4: Check data accessibility and verify access

Each dataset falls into one of the following accessibility categories. For each type, check the DAS and verify access as described.

### Open data
*Data is publicly available and included in the reproducibility package.*

- Check that the link in the DAS works and that the data actually exists at that location.
- You do not need to verify every value, but download the data to confirm that what is downloaded matches what is in the package.
- If the instructions for accessing the data are unclear, ask the authors to clarify.

### Accessible data
*Data is publicly available but does not allow redistribution and it is not included in the reproducibility package.*

- Follow the same steps as for Open data.
- Additionally, run the [Dataset Comparison Tool](https://github.com/worldbank/wb-reproducible-research-repository/tree/main/resources/compare_datasets) to compare the publicly available version with the data provided by the authors.
- If differences are due to periodic updates to the source data, note this in the verification report. If differences appear to be due to manual changes, confirm with the authors whether any modifications or processing were done. If processing was done in code, ask for the code. If it was done manually, ask the authors to document the steps. In general, encourage authors to start from the raw source data where possible.

### Limited-access data
*Data access requires purchase or human approval and is not included in the reproducibility package.*

- If the data is reasonably easy to access, follow the same steps as for Accessible data above.
- If access is difficult to obtain, document what was verified and flag any limitations in the report.

### Forthcoming data
*Data is forthcoming in the World Bank Microdata Library or Development Data Hub.*

- No access verification is needed at this stage.
- Add a note to the GitHub issue to return to this once the data is published: update the metadata entry with the published link, and if the data is Accessible or Limited-access, run the comparison tool at that point.

### Restricted data
*There is no documented public access path, or data access was granted directly to the authors under a custom license that does not allow redistribution; the data is not included in the reproducibility package.*

- Check that the DAS contains enough information for a user to understand where the data comes from and how it can be requested.



---

## Common issues to flag

- **No DAS at all.** Return the package; this is a blocker.
- **DAS covers only some datasets.** Use the hash tool to confirm all input datasets are listed.
- **Package starts from an intermediate file** (e.g., `cleaned_for_analysis.dta`). This is a blocker. Return to authors with an explanation of the policy.
- **Opening statement does not match the actual data** (e.g., claims all data is public when some is restricted). Flag for correction.
- **Statement of Rights is missing.** Required in every DAS.
- **URL is missing for publicly available data.** Required if the data is accessible online.
- **Access date is missing.** Required.
- **No license or redistribution statement for restricted data.** Required.
- **World Bank-owned data not deposited in MDL or DDH.** Flag and initiate deposit process (unless the exception applies).
- **Data only accessible via a staff member's personal storage.** Flag as a critical issue; data must be in a permanent, institutional location.
- **Restricted data has no access instructions.** A third party must be able to understand how to request the same data.

---

## Quick reference checklist

- [ ] DAS is present and clearly labeled
- [ ] Opening statement (if present) correctly reflects the data in the package
- [ ] Statement of Rights is included
- [ ] All input datasets are identified (use hash tool if needed)
- [ ] Each dataset is documented with required fields (filename, source, URL if available, access date, citation, license/redistribution terms, access instructions)
- [ ] License terms verified for each dataset
- [ ] World Bank-owned data is deposited in MDL or DDH, or flagged as forthcoming (with GitHub issue note)
- [ ] Data accessibility verified per category (link check, comparison tool where applicable)
- [ ] Restricted data has a documented access pathway and sufficient information for third parties
- [ ] Package starts from documented source data, not intermediate files
